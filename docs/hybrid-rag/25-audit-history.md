# Oblast 25 — Audit & change history (config + operace)

> Tato oblast pokrývá audit jako **základní platformovou schopnost** modulu `HybridRag` — ne samostatný modul (audit je platform capability `AuditInterceptor` zapisující per-modul `hybridrag_audit_entries`, viz CLAUDE.md §4 a poznámka "Audit is NOT a separate module"). Audit musí pokrýt DVA druhy záznamů: (a) **config-change audit** (každá změna `RagSetting` knobu — kdo, kdy UTC, old→new, scope) a (b) **operation audit** (search / answer / ingest / reindex / delete — actor, tenant, čas, query-text jako `[PersonalData]`, počet výsledků, `RetrievalStatus`, cost/tokeny, latence, applied preset/config-version, retrieved chunk ids). Vše slouží forenznímu "proč klientovi vyšla tahle odpověď".
>
> Frozen reuse seamy použité napříč oblastí: `AuditInterceptor` na `SaveChanges` → `hybridrag_audit_entries` (jen changed fields → JSONB); PII v auditu = `[PersonalData]` + `IDataSubject` → crypto-shred přes `PersonalDataProtector`; admin forensic read `GET /v1/hybridrag/admin/users/{id}/audit` (`PlatformPermissions.AuditRead`) až do erasure, pak `[erased]`; read pattern `GetProfileHandler.cs:12` (`IReadDbContextFactory`); outbox `RegisterUserHandler.cs:22`; 202 long work `StartDemoOperationHandler.cs:17` (`IOperationStore`); metriky `PlatformMetrics.Meter` (`platform.rag.*`); OTel GenAI spany = oblast 19; GDPR erasure retain audit = oblast 20.
>
> Klíčová invarianta této oblasti: **query-text a obsah dotazu/odpovědi NIKDY nesmí téct plaintextem do běžného logu ani do metrik** (jen hash + délka); plaintext drží POUZE audit, a to šifrovaně pod DEK subjektu. Po GDPR erasure se z auditního čtení stane `[erased]`, ale řádky se nemažou (AML/forenzní append-only).

## UC-25-01 — Audit změny konfiguračního knobu (RagSetting) přes AuditInterceptor
- **Actor / role:** tenant-admin (změna scope=Tenant) | platform-admin (změna scope=System default) | user (změna scope=User vlastního presetu)
- **Precondition:** Existuje entita `RagSetting` (snake_case tabulka `hybridrag_settings`, derived z `AuditableEntity`, `ITenantScoped`, scope sloupec `Scope = System|Tenant|Collection|User`, klíč `Key`, hodnota `Value` jako string/JSONB). Modul `Modules:HybridRag:Enabled=true`. Caller má permission `rag.manage` (`PlatformPermissions.RagManage`) pro Tenant/Collection scope. · **Trigger:** `PUT /v1/hybridrag/settings/{key}` (resp. `PATCH /v1/hybridrag/admin/settings/{key}` pro System) → `UpdateRagSettingCommand` → `UpdateRagSettingHandler`. · **Main flow:** (1) Handler načte tracked `RagSetting` z write kontextu (`IDbContextOutbox<HybridRagDbContext>`); (2) ověří permission + scope ownership (`ITenantContext.UserId`/`TenantId`); (3) přepíše `Value` (a `UpdatedBy`/`UpdatedAtUtc` z `IClock.UtcNow`); (4) `SaveChangesAndFlushMessagesAsync()` — při SaveChanges `AuditInterceptor` zachytí změněné property a zapíše do `hybridrag_audit_entries` jen DIFF (`{"Value":{"old":"hybrid","new":"vector_only"}}`) jako JSONB, spolu s `EntityType=RagSetting`, `EntityId`, `Action=Update`, `ActorUserId`, `TenantId`, `TimestampUtc`.
- **Postcondition / záruky:** V `hybridrag_audit_entries` existuje 1 append-only řádek s old→new diffem a actorem; změna nastavení je auditovatelná "kdo, kdy, z čeho na co". Audit běží automaticky — handler nepíše audit ručně. · **Tenancy / permissions:** Scope=Tenant → `ITenantScoped` global filter + RLS; actor z tokenu; `RagManage` permission gate na endpointu (`.RequirePermission(...)`). · **Reuse / canonical pattern:** `AuditInterceptor` na `SaveChanges`; write přes `RegisterUserHandler.cs:22` (outbox commit); permission gate jako `building-modularplatform-feature`. · **Data dotčena:** `hybridrag_settings`, `hybridrag_audit_entries`. · **Eventy:** volitelně `RagSettingChangedIntegrationEvent` (outbox) pro invalidaci cache presetů. · **Priorita:** P0

### Edge cases UC-25-01
- **EC-25-01-01 — Změna bez reálné diference (no-op write)** · Trigger: PUT se stejnou hodnotou `Value`. · Očekávané chování: EF nezaznamená žádnou modified property → `AuditInterceptor` NEZAPÍŠE řádek (no-op není audit-worthy). · Mechanismus: `AuditInterceptor` iteruje jen `EntityState.Modified` se skutečně změněnými `OriginalValue != CurrentValue`. · Severity: P2 · Test: integrační — 2× stejná hodnota → 0 audit řádků.
- **EC-25-01-02 — Změna více knobů v jedné transakci** · Trigger: bulk update presetu (`UpdateRagPresetCommand` mění 5 settings). · Očekávané chování: 5 samostatných audit řádků se SHODNÝM `TimestampUtc`/transaction id, každý se svým diffem. · Mechanismus: interceptor zpracuje každý tracked entity zvlášť v jednom `SaveChanges`. · Severity: P2 · Test: assert 5 řádků, sdílený čas.
- **EC-25-01-03 — Concurrent config change (poslední vyhrává) + pořadí auditu** · Trigger: dva admini mění stejný `Key` současně. · Očekávané chování: xmin + `ConcurrencyRetryBehavior` serializuje; vznikají DVA audit řádky ve správném kauzálním pořadí (druhý vidí old = výsledek prvního). · Mechanismus: xmin token na `RagSetting`; retry vyčistí tracker; audit po každém úspěšném commitu. · Severity: P1 · Test: 2-way concurrent, assert old₂ == new₁.
- **EC-25-01-04 — Změna config knobu mid-flight (běžící search ho nepoužije)** · Trigger: admin změní `retrieval.top_k` během probíhajícího answer flow. · Očekávané chování: běžící operace si snapshotla config-version na začátku (oblast 14 cost/preset capture), audit operace zaznamená STAROU verzi; nová verze platí až pro další request. · Mechanismus: operation audit ukládá `AppliedConfigVersion` (UC-25-14); config-change audit je nezávislý řádek. · Severity: P1 · Test: změna mid-flight → audit operace nese starou config-version.
- **EC-25-01-05 — Změna scope=System bez platform-admin permission** · Trigger: tenant-admin volá `PATCH /admin/settings/{key}`. · Očekávané chování: `403 ForbiddenException("rag.settings.scope_forbidden")`; audit selhání se zaznamená jako failed-operation (UC-25-10), NE jako úspěšná změna. · Mechanismus: permission gate + scope check před SaveChanges. · Severity: P0 · Test: 403 + 0 change audit řádků + 1 failed-auth audit.
- **EC-25-01-06 — Hodnota knobu obsahuje PII-like data omylem** · Trigger: admin uloží do `Value` text obsahující e-mail. · Očekávané chování: `RagSetting.Value` NENÍ `[PersonalData]` (config není subjektové PII) → audit ho drží plaintextem; validátor by měl odmítnout free-text PII v knob hodnotě (`rag.settings.value_invalid`). · Mechanismus: `UpdateRagSettingValidator` allowlist typů hodnot. · Severity: P2 · Test: e-mail v číselném knobu → validation error.
- **EC-25-01-07 — Změna na smazané/neexistující knob** · Trigger: PUT na `Key`, který neexistuje. · Očekávané chování: buď vytvoření (upsert → `Action=Create` audit) NEBO `404 NotFoundException("rag.settings.not_found")` dle politiky; rozhodnutí frozen v configu. · Mechanismus: handler rozhoduje create-vs-404. · Severity: P2 · Test: oba scénáře.
- **EC-25-01-08 — ExecuteUpdate na settings obejde audit** · Trigger: někdo refaktoruje hromadný reset knobů přes `db.RagSettings.Where(...).ExecuteUpdateAsync(...)`. · Očekávané chování: ZAKÁZÁNO pro auditované config změny — `ExecuteUpdate` bypassuje `AuditInterceptor` i xmin → ztráta auditu. Použít tracked mutaci. · Mechanismus: CLAUDE.md §4 caveat; code review / ArchUnitNET guard. · Severity: P0 · Test: review-level + assert že reset píše audit.
- **EC-25-01-09 — Audit řádek zachycuje actora i u admin-jménem-usera akce** · Trigger: platform-admin mění tenant setting jménem tenanta. · Očekávané chování: `ActorUserId` = skutečný admin (z tokenu), `TenantId` = cílový tenant; nikdy nezáměna actora za majitele scope. · Mechanismus: actor vždy z `ITenantContext.UserId` (Law 10). · Severity: P0 · Test: cross-tenant admin write → actor == admin id.

## UC-25-02 — Admin čte historii změn konfigurace ("kdo co kdy změnil")
- **Actor / role:** tenant-admin | platform-admin
- **Precondition:** Existují config audit řádky v `hybridrag_audit_entries` s `EntityType=RagSetting`. Caller má `rag.manage` nebo `PlatformPermissions.AuditRead`. · **Trigger:** `GET /v1/hybridrag/admin/settings/audit?key=&scope=&from=&to=&page=` → `GetSettingsAuditQuery` → `GetSettingsAuditHandler`. · **Main flow:** (1) read-only přes `IReadDbContextFactory` (`GetProfileHandler.cs:12`); (2) filtr `EntityType=RagSetting` + volitelně `Key`/`scope`/časové okno; (3) RLS + tenant filter omezí na vlastní tenant (platform-admin přes system kontext vidí napříč); (4) projekce na `SettingAuditEntryResponse` (key, old→new z JSONB diffu, actorUserId, actorDisplay, timestampUtc, scope); (5) `ApiResponse<Paged<...>>`.
- **Postcondition / záruky:** Admin vidí chronologickou, stránkovanou historii všech config změn; žádná mutace (query NIKDY neotevírá transakci). · **Tenancy / permissions:** RLS-scoped na tenant; `AuditRead`/`RagManage` gate; cizí tenant audit → prázdná stránka/404. · **Reuse / canonical pattern:** `GetProfileHandler.cs:12` (read factory), paging building-block. · **Data dotčena:** `hybridrag_audit_entries` (read-only). · **Eventy:** žádné. · **Priorita:** P1

### Edge cases UC-25-02
- **EC-25-02-01 — Paging stabilita při souběžných nových změnách** · Trigger: během stránkování přibývají nové audit řádky. · Očekávané chování: deterministické řazení `TimestampUtc DESC, Id DESC` (Guid v7 tie-break) → žádné přeskočení/duplikace. · Mechanismus: keyset/offset s Id tie-break. · Severity: P2 · Test: paging pod zápisem.
- **EC-25-02-02 — Filtr časového okna v UTC** · Trigger: `from/to` v lokálním čase klienta. · Očekávané chování: server interpretuje jako UTC (`DateTimeOffset`), žádný `DateTime.Now`. · Mechanismus: `IClock`/UTC konvence. · Severity: P2 · Test: hraniční sekunda okna.
- **EC-25-02-03 — Tenant-admin se snaží číst System-scope audit** · Trigger: filtr `scope=System`. · Očekávané chování: tenant-admin vidí jen Tenant/Collection/User scope svého tenanta; System default změny jen platform-admin. · Mechanismus: scope authorization v handleru. · Severity: P1 · Test: tenant-admin + scope=System → 403/prázdné.
- **EC-25-02-04 — Diff JSONB s null old (vznik knobu)** · Trigger: `Action=Create` řádek. · Očekávané chování: `old=null, new=...` korektně serializováno; UI rozliší create vs update. · Mechanismus: interceptor zapíše null pro neexistující original. · Severity: P3 · Test: create → old null.
- **EC-25-02-05 — Velký objem config historie (rok provozu)** · Trigger: dotaz bez filtru. · Očekávané chování: povinný default page-size cap + index na (`EntityType`,`TimestampUtc`); žádný full scan. · Mechanismus: paging cap + DB index. · Severity: P2 · Test: 100k řádků → indexed paged dotaz < limit.
- **EC-25-02-06 — Actor display name po smazaném userovi** · Trigger: actor admin byl mezitím GDPR-erased. · Očekávané chování: `actorUserId` zůstává (forenzní), display name → `[erased]`/anonymizováno, řádek se NEMAŽE. · Mechanismus: oblast 20 retain audit; join na Identity vrací anonymizovaný název. · Severity: P1 · Test: erased actor → id zachováno, jméno `[erased]`.
- **EC-25-02-07 — Read query omylem otevře transakci** · Trigger: code review. · Očekávané chování: zakázáno (Law 2 — queries NEVER open transaction). · Mechanismus: read factory bez SaveChanges. · Severity: P1 · Test: ArchUnitNET/review.

## UC-25-03 — Audit operace search/answer (forenzní "proč vyšla tahle odpověď")
- **Actor / role:** user (vlastní dotaz) | system/worker (interní re-rank job)
- **Precondition:** Modul aktivní; existuje `RagConversation`/`RagTurn`/`AnswerCitation`/`RagTrace`. Search/answer endpoint vrací odpověď. · **Trigger:** `POST /v1/hybridrag/search` nebo `POST /v1/hybridrag/answer` → `RunRagAnswerCommand`/`RunRagSearchHandler`. · **Main flow:** (1) handler provede retrieval (vektor + graf + BM25) a generaci; (2) na konci zapíše operation-audit záznam: `Action=Search|Answer`, `ActorUserId`, `TenantId`, `TimestampUtc`, `QueryText` (`[PersonalData]`, šifrované), `ResultCount`, `RetrievalStatus` (Ok|Degraded|Empty|Failed), `TokensIn/TokensOut`, `CostMicros`, `LatencyMs`, `AppliedPreset`, `AppliedConfigVersion`, `RetrievedChunkIds` (JSONB pole id, NE obsah), `AnswerCitationIds`; (3) `SaveChangesAndFlushMessagesAsync()` → `AuditInterceptor` + perzistence trace. Operation-audit pro mutaci/answer = **full** (ne sampling).
- **Postcondition / záruky:** Pro každý answer existuje forenzní stopa stačící rekonstruovat "proč tahle odpověď": jaký dotaz (šifrovaně), s jakým presetem/config-version, které chunky se vrátily, kolik to stálo, jak dlouho to trvalo, jaký status. · **Tenancy / permissions:** owner z tokenu; `IUserOwned` → RLS; čtení jen vlastní/admin forensic. · **Reuse / canonical pattern:** audit přes `AuditInterceptor`; `[PersonalData]` na `QueryText`; metriky `platform.rag.*`; trace = oblast 19. · **Data dotčena:** `hybridrag_audit_entries`, `hybridrag_rag_turns`, `hybridrag_rag_traces`, `hybridrag_answer_citations`. · **Eventy:** žádné (audit je side-effect commitu). · **Priorita:** P0

### Edge cases UC-25-03
- **EC-25-03-01 — Query-text leak do plain logu/metriky** · Trigger: developer přidá `LogInformation("query={query}", queryText)`. · Očekávané chování: ZAKÁZÁNO — do logu/metrik jen `QueryHash` (SHA-256) + `QueryLength`; plaintext jen do šifrovaného auditu. · Mechanismus: §4 PII konvence; lint/review pravidlo; metrika `platform.rag.search` má jen hash/length dimenze. · Severity: P0 · Test: log scrape assert žádný plaintext; metrika bez raw query.
- **EC-25-03-02 — Answer selhal (LLM timeout) — degraded audit** · Trigger: generace vyhodí timeout, retrieval ale proběhl. · Očekávané chování: audit řádek s `RetrievalStatus=Degraded/Failed`, `ResultCount` z retrievalu, `TokensOut=0`, error kategorie; operace je auditovaná i při selhání (UC-25-10). · Mechanismus: handler zapíše audit v finally/catch větvi před propagací chyby. · Severity: P0 · Test: vynucený timeout → failed audit existuje.
- **EC-25-03-03 — Prázdný retrieval (0 chunků)** · Trigger: dotaz mimo korpus. · Očekávané chování: `RetrievalStatus=Empty`, `ResultCount=0`, `RetrievedChunkIds=[]`; audit přesto vznikne (forenzně cenné "nic se nenašlo"). · Mechanismus: full audit i pro empty. · Severity: P1 · Test: out-of-corpus dotaz → empty audit.
- **EC-25-03-04 — Velmi dlouhý query-text** · Trigger: 50k znaků dotaz. · Očekávané chování: validátor cap (`rag.query.too_long`) PŘED zpracováním; pokud projde, audit šifruje plnou délku, log jen length. · Mechanismus: `RunRagAnswerValidator` length cap. · Severity: P2 · Test: nadlimitní dotaz → validation error.
- **EC-25-03-05 — RetrievedChunkIds odkazují na chunky jiného usera/tenanta?** · Trigger: forenzní kontrola izolace. · Očekávané chování: nikdy — retrieval je RLS/scope-omezený na Tenant|User korpus; chunk ids v auditu pochází jen z dosažitelného korpusu. · Mechanismus: dvouvrstvý korpus Scope + RLS. · Severity: P0 · Test: cross-tenant izolace na úrovni retrievalu.
- **EC-25-03-06 — Cost/tokeny nezachyceny při streamované odpovědi** · Trigger: SSE streaming answer, klient disconnectne. · Očekávané chování: audit se zapíše s tokeny spočtenými do bodu přerušení + `RetrievalStatus` značící partial; disconnect-safe save. · Mechanismus: handler ukládá audit i při disconnectu (jako vibe streaming pattern z memory). · Severity: P1 · Test: disconnect mid-stream → audit s partial cost.
- **EC-25-03-07 — Sampling omylem aplikovaný na answer audit** · Trigger: config sampling rate < 100% (UC-25-11). · Očekávané chování: sampling se NIKDY nevztahuje na answer/mutace/admin — jen na high-volume read search. Answer = vždy full. · Mechanismus: audit policy rozlišuje `Action`. · Severity: P0 · Test: sampling=1% → answer stále 100% auditovaný.
- **EC-25-03-08 — Idempotentní retry answeru (stejný request)** · Trigger: klient zopakuje request s idempotency key. · Očekávané chování: druhý pokus nezdvojuje audit pokud vrací cached výsledek; pokud reálně přepočítá, vznikne nový audit (každé reálné provedení = řádek). · Mechanismus: UNIQUE idempotency key + catch `DbUpdateException`. · Severity: P2 · Test: idempotentní replay → bez duplicitního auditu cached cesty.
- **EC-25-03-09 — Audit zachytí applied preset i při fallbacku** · Trigger: vybraný preset neexistuje → fallback na default. · Očekávané chování: audit zaznamená SKUTEČNĚ použitý preset (default) + flag že došlo k fallbacku, ne požadovaný. · Mechanismus: handler loguje effective config. · Severity: P1 · Test: neexistující preset → audit nese effective default.

## UC-25-04 — Audit ingest / reindex operace (long-running, 202)
- **Actor / role:** user (ingest do User scope) | tenant-admin (ingest do Tenant korpusu) | system/worker (saga stages)
- **Precondition:** Existuje `Document`, `Chunk`, `IngestSaga`, `IOperationStore`. Ingest běží přes 202 + worker. · **Trigger:** `POST /v1/hybridrag/documents` (upload→ingest) nebo `POST /v1/hybridrag/collections/{id}/reindex` → 202 + `Location: /v1/hybridrag/operations/{id}`; worker spouští `IngestSaga`. · **Main flow:** (1) accept handler vytvoří operation (`StartDemoOperationHandler.cs:17` pattern) + audit `Action=IngestRequested` (actor=user); (2) worker stages (extract→chunk→embed→graph-extract→index) každý zapíše stage audit s `actor=system`, `IngestSagaId`, stage název, počet chunků/nodů, latence; (3) terminální stav (`Completed|Failed`) → finální audit s totals (documents, chunks, tokens embed cost). · **Postcondition / záruky:** Kompletní auditní stopa ingestu od požadavku po dokončení, včetně každé saga stage a jejího actora (user vs system); cost embeddingu zachycen. · **Tenancy / permissions:** operation `IUserOwned`→RLS; stage audit `actor=system` ale `TenantId`/`OnBehalfOfUserId` zachován. · **Reuse / canonical pattern:** `StartDemoOperationHandler.cs:17` (202+operation), saga `CreditPurchaseSaga`, outbox `RegisterUserHandler.cs:22`. · **Data dotčena:** `hybridrag_documents`, `hybridrag_chunks`, `hybridrag_ingest_sagas`, `operations`, `hybridrag_audit_entries`. · **Eventy:** `DocumentIngestedIntegrationEvent`/saga messages. · **Priorita:** P1

### Edge cases UC-25-04
- **EC-25-04-01 — Saga stage selže a retry-uje (Wolverine retry/DLQ)** · Trigger: embed stage hodí transient chybu. · Očekávané chování: každý reálný pokus = audit řádek `actor=system, stage=embed, attempt=n, status=Retrying/Failed`; inbox dedup nezpůsobí ztrátu, ale ani falešné zdvojení completed stavu. · Mechanismus: Wolverine retry-with-cooldown + DLQ; idempotentní stage. · Severity: P1 · Test: vynucený transient → audit ukazuje pokusy.
- **EC-25-04-02 — Actor u worker stage je system, ne původní user** · Trigger: forenzní kontrola. · Očekávané chování: `ActorUserId` = system sentinel (nebo null + `ActorKind=System`), `OnBehalfOfUserId` = původce ingestu; nikdy nezáměna. · Mechanismus: `SystemTenantContext`/system actor v worker hostu. · Severity: P1 · Test: stage audit má ActorKind=System + OnBehalfOf.
- **EC-25-04-03 — ExecuteUpdate při bulk reindexu obejde audit chunků** · Trigger: reindex přepíše embeddings přes `ExecuteUpdate` (pgvector carve-out). · Očekávané chování: vektorové sloupce nejsou audit-worthy per-row (objem) → audituje se SOUHRN reindexu (1 operation audit s počty), ne každý chunk; vědomé rozhodnutí dokumentované. · Mechanismus: §4 ExecuteUpdate bypass caveat + summary audit. · Severity: P1 · Test: reindex → 1 summary audit, ne N chunk řádků.
- **EC-25-04-04 — Reindex spuštěn nad cizí kolekcí** · Trigger: user reindexuje `collectionId` jiného tenanta. · Očekávané chování: `404` (RLS-scoped cizí id → not found), failed-operation audit; žádné stage audity nevzniknou. · Mechanismus: RLS na `KnowledgeCollection`. · Severity: P0 · Test: cizí id → 404 + 0 stage audit.
- **EC-25-04-05 — Document obsahuje PII v textu** · Trigger: ingest dokumentu s osobními údaji. · Očekávané chování: audit ingestu NEUKLÁDÁ obsah dokumentu (jen metadata/počty); obsah žije v `hybridrag_documents`/`chunks` (samostatná oblast PII at rest), audit drží jen `DocumentId`+hash. · Mechanismus: audit zachytává metadata, ne payload. · Severity: P1 · Test: ingest → audit bez plaintextu obsahu.
- **EC-25-04-06 — Saga timeout (abandon)** · Trigger: embed provider dlouho nedostupný → saga `TimeoutMessage`. · Očekávané chování: finální audit `status=Abandoned/Failed` s důvodem; operation→Failed; reconcile job potvrdí. · Mechanismus: saga timeout + `ReconcileStaleOperationsCommand`. · Severity: P1 · Test: vynucený timeout → abandoned audit.
- **EC-25-04-07 — Duplicitní ingest téhož dokumentu (idempotency)** · Trigger: upload stejného content-hash. · Očekávané chování: idempotency UNIQUE → catch `DbUpdateException`, audit `Action=IngestDeduplicated` místo nového ingestu. · Mechanismus: UNIQUE content-hash key. · Severity: P2 · Test: dvojí upload → dedup audit.
- **EC-25-04-08 — Cost embeddingu nezachycen při partial failure** · Trigger: embed 800/1000 chunků pak fail. · Očekávané chování: audit zaznamená cost za 800 skutečně provedených (ne 0, ne 1000); forenzně přesné. · Mechanismus: stage audit akumuluje průběžně. · Severity: P1 · Test: partial fail → cost == provedené.

## UC-25-05 — Audit delete operací (collection / document / chunk / konverzace)
- **Actor / role:** user (smazání vlastních dat) | tenant-admin (smazání tenant kolekce) | system/worker (GDPR erasure fan-out)
