# Oblast 11 — Entity resolution
Tato oblast pokrývá rozpoznání, sjednocení a údržbu kanonických identit grafových uzlů (`GraphNode`) napříč ingestovanými dokumenty — blocking (redukce n²), similarity matching, LLM adjudikaci na hraničním pásmu, kanonický merge s `EntityAlias`, manual review queue a stabilitu kanonického ID. Mapuje se na build fázi „grafová vrstva + offline batch joby" (po extrakci entit z chunků, před community detection). Je to **kritický bod přesnosti grafu**: over-merge zničí precision (dvě různé firmy splynou v jeden uzel), under-merge roztříští graf na ostrovy (ABC Motors / ABC Motors LLC / ABC  Motors zůstanou 3 uzly a 1-2 hop traverz je nenajde). Default je **opatrný merge (precision > recall)**, hraniční páry jdou do review queue, kanonické ID se nikdy nemění in-place a merge je idempotentní.

## UC-11-01 — Blocking (generování kandidátních párů)
- **Actor / role:** system/worker (offline batch job, krok ingest saga nebo samostatný resolution job)
- **Precondition:** existuje ≥2 `GraphNode` se stejným `Scope` + `TenantId` (a u `User` scope stejným `OwnerUserId`); každý má vyplněný `CanonicalKey` (normalizovaný klíč) a `PropsJson` (z něj se odvodí název/aliasy); embeddingy názvů uzlů jsou spočítané (sdílí pipeline s chunk embeddingem, OpenAI text-embedding-3-large, 3072 dim) nebo jsou součástí dedikovaného name-embedding sloupce.
- **Trigger:** Wolverine durable command `BuildBlockingCandidatesCommand` (krok ingest saga `IngestSaga` po extrakci entit) nebo cron `ResolveEntitiesJob` (Jobs host) → `dispatcher.Send(BuildBlockingCandidatesCommand)`.
- **Main flow:** endpoint/job → dispatcher → `BuildBlockingCandidatesHandler` → načte nové/dirty uzly přes `IReadDbContextFactory` → vytvoří blocking klíče dvěma kanály: (a) **exact/normalized-key block** = LINQ `GroupBy(n => n.CanonicalKey)` (přesný a prefixový/n-gram klíč), (b) **ANN block** = pro každý uzel pgvector `OrderBy(n => n.NameEmbedding.CosineDistance(seed))` přes EF/LINQ, top-K nejbližších (K z konfigurace, default ~25). Sjednotí kandidáty z obou kanálů, deduplikuje páry `(min(idA,idB), max(idA,idB))`, vyloučí už-mergnuté (stejné `CanonicalNodeId`) a self-páry → vrací kandidátní páry pro UC-11-02. Žádný plný n² scan.
- **Postcondition / záruky:** žádná mutace grafu (blocking je čtení + příprava); kandidátní páry předány dál v rámci saga kroku nebo zapsány do dočasné pracovní množiny; idempotentní (běh přes stejné uzly dá stejnou množinu); deterministicky uspořádané páry (kanonické pořadí id).
- **Tenancy / permissions:** Scope Tenant|User — blocking běží VŽDY uvnitř jedné izolační hranice (nikdy nemíchá kandidáty napříč tenanty ani napříč privátními uzly různých uživatelů); worker běží jako system kontext (`SystemTenantContext`) ale dotazy filtruje explicitně na konkrétní `TenantId`/`Scope`/`OwnerUserId` zpracovávané dávky; RLS na `graph_nodes` je defence-in-depth.
- **Reuse / canonical pattern:** read query `GetProfileHandler.cs:12` (IReadDbContextFactory); ANN dotaz = vektorový retrieval pattern modulu (CosineDistance LINQ); durable saga krok = `CreditPurchaseSaga.cs:30`.
- **Data dotčena:** `graph_nodes` (čtení), dočasná pracovní množina párů · **Eventy:** žádný (interní krok)
- **Priorita:** P0

### Edge cases UC-11-01
- **EC-11-01-01 — Supernode generický název** · Trigger: uzel s `CanonicalKey="company"` / „Inc" / „GmbH" se v ANN bloku spojí s tisíci jiných · Očekávané chování: zavést stop-list generických tokenů + cap na velikost bloku (např. max 200 kandidátů/uzel), nadlimit → uzel označit jako „needs manual review", NE generovat kvadratické množství párů · Mechanismus: blocking handler aplikuje token stop-list + per-block cap; zákon „graceful degradation = explicit Degraded flag" → uzel dostane `ResolutionStatus=Deferred` místo tichého vynechání · Severity: P0 · Test: integrační — vytvoř 5000 uzlů sdílejících token „GmbH", assert počet vygenerovaných párů ≤ cap a uzly mají Deferred flag
- **EC-11-01-02 — Prázdný `CanonicalKey`** · Trigger: extrakce dala uzel bez názvu (jen typ) · Očekávané chování: uzel se NEúčastní normalized-key bloku (jinak by se všechny prázdné slily); zařadí se jen do ANN bloku, a pokud i embedding chybí → vyloučit z resolution + log WARN · Mechanismus: validace v handleru, `Where(n => n.CanonicalKey != "")` · Severity: P1 · Test: unit — uzel s prázdným klíčem negeneruje exact-block páry
- **EC-11-01-03 — Embedding chybí / dimension drift** · Trigger: uzel má `NameEmbedding == null` nebo jiný počet dimenzí než 3072 (model drift) · Očekávané chování: ANN blok ho přeskočí, fallback jen na normalized-key blok; dimension mismatch = fail-fast log + uzel `Deferred`, NIKDY tiše porovnat různě-dimenzionální vektory · Mechanismus: guard na dimenzi před CosineDistance; zákon „embedding dimension/model drift" z taxonomie · Severity: P0 · Test: integrační — vlož uzel s 1536-dim vektorem, assert nehavaruje a uzel skončí Deferred
- **EC-11-01-04 — Concurrent ingest přidá uzly během blockingu** · Trigger: druhý dokument se indexuje paralelně a vytvoří nové uzly · Očekávané chování: blocking pracuje nad snapshotem dirty-setu; nové uzly chytne další běh (job je opakovatelný); žádný lock na celý graf · Mechanismus: dirty-flag / `ResolutionStatus=Pending` na uzlech, handler bere jen Pending; idempotence · Severity: P1 · Test: integrační — spusť 2 blocking běhy paralelně, assert žádný duplicitní pár ani crash
- **EC-11-01-05 — Cross-scope kontaminace** · Trigger: tenant uzel a privátní user uzel mají stejný `CanonicalKey` · Očekávané chování: NIKDY nevytvořit kandidátní pár přes hranici Scope (tenant vs user-private); blocking partitionuje podle `(Scope, OwnerUserId)` · Mechanismus: zákon per-user privacy within tenant; partition klíč zahrnuje Scope · Severity: P0 · Test: integrační — stejný název v tenant + user scope, assert 0 cross-scope párů
- **EC-11-01-06 — Cizí jazyk / encoding názvu** · Trigger: „Müller" vs „Mueller" vs „Muller", azbuka, CJK · Očekávané chování: normalizace `CanonicalKey` musí být Unicode-aware (NFKC fold, diakritika, case-fold) přes battle-tested knihovnu, ne ad-hoc regex; blocking pak najde varianty · Mechanismus: normalizační funkce sdílená s `EntityAlias.NormalizedKey`; zákon „battle-tested před vlastní implementací" · Severity: P1 · Test: unit — „Müller"/„Mueller" produkují stejný/blízký blok
- **EC-11-01-07 — Self-pár a už-mergnuté uzly** · Trigger: ANN vrátí uzel sám sebe nebo dva už sjednocené uzly (stejný `CanonicalNodeId`) · Očekávané chování: odfiltrovat self i už-mergnuté páry před matchingem · Mechanismus: `Where(p => p.A != p.B && a.CanonicalNodeId != b.CanonicalNodeId)` · Severity: P2 · Test: unit — už-mergnutý pár se negeneruje

## UC-11-02 — Similarity matching (skórování kandidátních párů)
- **Actor / role:** system/worker
- **Precondition:** existuje množina kandidátních párů z UC-11-01; oba uzly mají název a (volitelně) name-embedding.
- **Trigger:** durable command `ScoreCandidatePairsCommand` (navazuje na blocking v sáze nebo job).
- **Main flow:** handler pro každý pár spočítá kompozitní skóre v C#: (a) **embedding cosine similarity** = `1 - CosineDistance` (z pgvektoru, LINQ), (b) **Jaro-Winkler** na surové/normalizované názvy (string knihovna, ne raw SQL), (c) typová shoda (`GraphNode.Type` musí sedět — firma×osoba se neslučuje), (d) shoda strukturovaných propů z `PropsJson` (IČO/DUNS/email pokud jsou → silný deterministický signál). Kombinuje do skóre `s ∈ [0,1]` a zařadí pár do pásma: **auto-merge** (`s ≥ high`, default ~0.92, NEBO deterministický identifikátor shodný), **ambiguous** (`low ≤ s < high`, ~0.75–0.92 → UC-11-03 LLM), **no-match** (`s < low`, ~0.75 → zahodit). Prahy z konfigurace `Rag:EntityResolution:*`.
- **Postcondition / záruky:** každý pár má skóre + zařazení do pásma; žádná mutace grafu; deterministické (stejné vstupy → stejné skóre, žádná závislost na pořadí); typově nekompatibilní páry odfiltrovány.
- **Tenancy / permissions:** Scope Tenant|User stejně jako blocking; čistá výpočetní fáze, žádná RLS-citlivá mutace.
- **Reuse / canonical pattern:** vektorová similarita = retrieval pattern modulu (CosineDistance LINQ); kompozitní RRF/skórování v C# = zákon „RRF/scoring v C#, ne raw SQL"; read přes `GetProfileHandler.cs:12`.
- **Data dotčena:** `graph_nodes` (čtení), pracovní množina skóre · **Eventy:** žádný
- **Priorita:** P0

### Edge cases UC-11-02
- **EC-11-02-01 — Míchání raw skóre z různých metrik** · Trigger: naivní průměr cosine (0–1) a Jaro-Winkler (0–1) bez kalibrace, nebo míchání raw embedding score a BM25-like score · Očekávané chování: každý signál normalizovat/kalibrovat na společné měřítko PŘED kombinací; míchání raw skóre = bug (stejně jako RRF v retrievalu) · Mechanismus: explicitní váhy + kalibrace v C#; analogie zákona „RRF correctness — raw-score míchání = bug" · Severity: P0 · Test: unit — pár s vysokým cosine ale zcela jiným názvem nesmí dostat auto-merge
- **EC-11-02-02 — Typová neshoda přebije podobnost** · Trigger: osoba „Apple" (příjmení) vs firma „Apple Inc." · Očekávané chování: rozdílný `GraphNode.Type` → tvrdé no-match nebo silná penalizace, NIKDY auto-merge napříč typy · Mechanismus: typový guard má veto · Severity: P0 · Test: unit — person×company pár nikdy nedostane ≥ high skóre
- **EC-11-02-03 — Deterministický identifikátor vs fuzzy názvy** · Trigger: dva uzly s odlišnými názvy ale shodným IČO/DUNS/registračním číslem v `PropsJson` · Očekávané chování: shoda silného deterministického klíče → auto-merge i při nízké název-podobnosti (přepíše fuzzy); naopak různá IČO → tvrdé no-match i při shodném názvu (dvě „ABC s.r.o." s jiným IČO jsou různé firmy) · Mechanismus: deterministické signály mají prioritu nad fuzzy v kombinační logice · Severity: P0 · Test: integrační — stejný název + různé IČO ⇒ NEsloučí (precision)
- **EC-11-02-04 — Akronymy a zkratky** · Trigger: „IBM" vs „International Business Machines" · Očekávané chování: nízké Jaro-Winkler + možná střední embedding → spadne do ambiguous pásma → LLM adjudikace (UC-11-03), NE tiché no-match ani auto-merge · Mechanismus: pásmové zařazení; embedding zachytí sémantiku · Severity: P1 · Test: unit — akronym↔plný název → ambiguous
- **EC-11-02-05 — Numerické/datumové entity** · Trigger: uzly „Q1 2024" vs „Q1 2025" mají vysokou string i embedding podobnost ale jsou různé · Očekávané chování: u temporálních/numerických typů zvýšit práh / vyžadovat exact match na hodnotě, ne fuzzy · Mechanismus: typově-specifická pravidla skórování · Severity: P2 · Test: unit — dvě různá čtvrtletí se neslučují
- **EC-11-02-06 — Whitespace / interpunkce („ABC  Motors")** · Trigger: dvojitá mezera, tečky, pomlčky · Očekávané chování: normalizace před Jaro-Winkler (collapse whitespace, strip punctuation) → vysoké skóre → auto-merge stejné firmy · Mechanismus: sdílená normalizace s `NormalizedKey` · Severity: P1 · Test: unit — „ABC  Motors" a „ABC Motors" → auto-merge
- **EC-11-02-07 — Order-independence skórování** · Trigger: pár `(A,B)` vs `(B,A)` · Očekávané chování: symetrické skóre (Jaro-Winkler i cosine jsou symetrické; kombinace nesmí záviset na pořadí) · Mechanismus: kanonické pořadí + symetrické metriky; zákon „handlery order-independent" · Severity: P1 · Test: unit — `score(A,B) == score(B,A)`
- **EC-11-02-08 — Prázdný kandidát / null embedding na jedné straně** · Trigger: jeden uzel bez embeddingu · Očekávané chování: degradovat na jen-string skóre, NE havarovat ani implicitně 0; označit pár jako „partial signal" · Mechanismus: null-guard, explicit Partial flag · Severity: P2 · Test: unit — pár s jedním null embeddingem skóruje jen Jaro-Winkler

## UC-11-03 — LLM adjudikace ambiguous pásma
- **Actor / role:** system/worker (Claude přes `IChatClient` / Anthropic.SDK)
- **Precondition:** existuje pár v ambiguous pásmu (~0.75–0.92) z UC-11-02; chat provider dostupný (nebo fake pod `Rag:UseFakeGateways`).
- **Trigger:** durable command `AdjudicateAmbiguousPairCommand` (per pár nebo dávka), dispatchnutý ze skórovacího kroku.
- **Main flow:** handler sestaví prompt s názvy + relevantními propy + krátkým kontextem (snippet z dokumentu, kde se entita vyskytla) → zavolá Claude přes `IChatClient` (tool-free strukturovaný výstup: `{ same_entity: bool, confidence: float, reason: string }`) → na základě verdiktu + confidence pár přeřadí na auto-merge (UC-11-04) nebo no-match, nebo (při nízké LLM confidence / konfliktu se signály) do **manual review queue** (UC-11-05). Prompt-cache: statický systémový prefix (instrukce + few-shot) první, volatilní (konkrétní pár) za breakpointem.
- **Postcondition / záruky:** každý adjudikovaný pár má LLM verdikt + reason uložený pro audit/review; rozhodnutí je vstup do merge, NE přímá mutace grafu; idempotentní (re-adjudikace téhož páru přepíše verdikt, nemění kanonické ID).
- **Tenancy / permissions:** Scope Tenant|User; LLM dostává JEN data uvnitř izolační hranice (nikdy nemíchá entity dvou tenantů do jednoho promptu); tenant/user id se bere z kontextu dávky, NIKDY z LLM výstupu.
- **Reuse / canonical pattern:** `ClaudeVibeAgentGateway.cs:85` (IChatClient + Anthropic.SDK), durable 202+worker = `SendMessageHandler.cs:45` → `ProcessVibeTurnCommand.cs:84`; fake pod flagem = `MarketingModule.cs:51`.
- **Data dotčena:** `graph_nodes` (čtení), adjudikační záznam (verdikt+reason, případně na review entitě) · **Eventy:** žádný (interní)
- **Priorita:** P1

## UC-11-04 — Kanonický merge + EntityAlias
- **Actor / role:** system/worker
- **Precondition:** pár byl klasifikován jako auto-merge (UC-11-02 high pásmo / deterministický klíč / LLM „same" s dostatečnou confidence).
- **Trigger:** durable command `MergeEntitiesCommand(WinnerNodeId, LoserNodeId, Reason)` přes outbox.
- **Main flow:** handler vybere **kanonický (winner) uzel** deterministicky (pravidlo: nejstarší `CreatedAt` / nejnižší Guid v7 — stabilní, nezávislé na pořadí) → na `IDbContextOutbox` v jedné transakci: (1) vytvoří `EntityAlias` řádky (`CanonicalNodeId=winner`, `RawValue` = název loseru i jeho existující aliasy, `NormalizedKey`), (2) přesměruje hrany — `GraphEdge.SourceNodeId/TargetNodeId == loser` přepíše na winner (přes tracked entity / atomic guard), (3) loser uzel označí jako sloučený (`MergedIntoId=winner`, ne fyzický delete — kvůli stabilitě ID a auditu), (4) publikuje `EntitiesMergedIntegrationEvent` → `SaveChangesAndFlushMessagesAsync` (= commit). Idempotence: UNIQUE `(CanonicalNodeId, NormalizedKey)` na `EntityAlias` + check `MergedIntoId` (už sloučený loser → no-op).
- **Postcondition / záruky:** winner zůstává stabilní kanonické ID; loser ukazuje na winner; všechny hrany loseru přesměrované; `EntityAlias` zachycuje historické názvy; event publikován; opakovaný merge téhož páru = no-op (idempotentní). Audit zachytí změnu (interceptor na SaveChanges).
- **Tenancy / permissions:** Scope Tenant|User; merge JEN uvnitř izolační hranice (winner i loser stejný `(Scope, TenantId, OwnerUserId)`); RLS na `graph_nodes`/`graph_edges`/`entity_aliases`.
- **Reuse / canonical pattern:** outbox publish + idempotency catch `DbUpdateException` = `RegisterUserHandler.cs:22`; přesměrování hran tracked + xmin = `ConcurrencyRetryBehavior`; atomic guard pattern jako Billing debit.
- **Data dotčena:** `graph_nodes`, `graph_edges`, `entity_aliases` · **Eventy:** `EntitiesMergedIntegrationEvent`
- **Priorita:** P0

### Edge cases UC-11-04
- **EC-11-04-01 — Idempotentní opakovaný merge** · Trigger: stejný `(winner,loser)` přijde dvakrát (double delivery / retry) · Očekávané chování: druhý běh je no-op (loser už `MergedIntoId=winner`, alias UNIQUE konflikt → catch `DbUpdateException` → vrať existující stav) · Mechanismus: inbox dedup (UNIQUE MessageId) + UNIQUE alias key + `MergedIntoId` guard; `RegisterUserHandler.cs:22` · Severity: P0 · Test: integrační — dvojí merge command ⇒ 1 alias sada, 0 duplicit
