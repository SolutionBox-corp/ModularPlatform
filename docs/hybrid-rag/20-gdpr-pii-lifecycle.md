# Oblast 20 — GDPR / PII / data lifecycle
Pokrývá veškerou ochranu osobních údajů a životní cyklus dat v modulu **HybridRag**: export a výmaz osobních dat napříč chunky/dokumenty/embeddingy/grafem, krypto-shred DEK, šifrování PII v klidu na `Chunk.Content` a `GraphNode.PropsJson`, anonymizaci append-only auditu, retenční sweep, omezení životnosti PII v durable obálkách a obranu proti nepřímé prompt-injection na ingestu (untrusted dokumenty = supply-chain povrch). Mapuje se na build fázi **„HybridRag — GDPR & data-lifecycle hardening"** (poslední fáze před GA modulu), která navazuje na existující platformní GDPR fan-out (`UserErasureRequested` → `IErasePersonalData` per modul) a krypto-shredder z modulu Gdpr. Modul HybridRag MUSÍ zaregistrovat svoje port-implementace, jinak GDPR fan-out tento modul tiše přeskočí.

## UC-20-01 — Export osobních dat uživatele (IExportPersonalData)
- **Actor / role:** system/worker (spuštěno Gdpr modulem v rámci `UserDataExportRequested`)
- **Precondition:** Uživatel `U` má v HybridRag aspoň jeden `Document`/`Chunk`/`GraphNode` se `Scope=User` a `OwnerUserId=U`. Modul má v `RegisterServices` zaregistrováno `AddScoped<IExportPersonalData, HybridRagPersonalDataExporter>()`.
- **Trigger:** Gdpr fan-out volá `IExportPersonalData.ExportAsync(userId, ct)` (Worker host).
- **Main flow:** Gdpr export koordinátor → dispatch interního `ExportHybridRagDataQuery(userId)` → handler otevře read kontext přes `IReadDbContextFactory` → načte všechny `Document` + `Chunk` + `GraphNode`/`GraphEdge`/`EntityAlias` kde `OwnerUserId == userId && Scope == User` (LINQ, ne raw SQL) → dešifruje `[Encrypted]` pole přes model-level konvertor (read factory umí dešifrovat dokud DEK žije) → složí strukturovaný DTO (kolekce → dokumenty → chunky text/contextual prefix → entity grafu jichž je subjekt) → vrátí jako součást agregovaného exportního balíku.
- **Postcondition / záruky:** Read-only operace, žádná mutace, žádný event. Export obsahuje VŠECHNA user-owned data subjektu vč. dešifrovaného textu chunků; tenant-scoped korpus (Scope=Tenant) se NEexportuje do osobního exportu jednotlivce (není jeho osobní údaj, je firemní).
- **Tenancy / permissions:** Scope=User řez; běží v SYSTEM kontextu (worker), takže RLS per-user filtr neaplikuje — handler MUSÍ explicitně filtrovat `OwnerUserId == userId` (system kontext vidí vše). Žádná HTTP permission (interní port).
- **Reuse / canonical pattern:** `GetProfileHandler.cs:12` (read přes `IReadDbContextFactory`) + Billing/Notifications `IExportPersonalData` impl; registrace dle CLAUDE.md §9b „GDPR fan-out works only if PII modules register their port impls".
- **Data dotčená:** `documents`, `chunks`, `graph_nodes`, `graph_edges`, `entity_aliases` · **Eventy:** žádné (query)
- **Priorita:** P0

### Edge cases UC-20-01
- **EC-20-01-01 — Modul nezaregistroval exporter** · Trigger: `HybridRagPersonalDataExporter` chybí v DI · Očekávané chování: fan-out tiše přeskočí modul a export uživatele je NEÚPLNÝ (GDPR porušení) · Mechanismus: ArchUnitNET/boot-test pravidlo MUSÍ vynutit, že modul implementující entity s `IDataSubject` má registrovaný `IExportPersonalData` (analogie pravidla `[PersonalData]`↔`IDataSubject`); fail-fast v `Hosts.Tests` · Severity: P0 · Test: boot test — assert `IExportPersonalData` resolv. obsahuje HybridRag impl.
- **EC-20-01-02 — Uživatel nemá žádná data** · Trigger: `U` nikdy nic neingestoval · Očekávané chování: vrátí prázdnou, ale validní strukturu (ne null, ne výjimka) · Mechanismus: handler vrací prázdné kolekce; export koordinátor mergne · Severity: P2 · Test: integrační — export user bez dat → 200 s prázdnou HybridRag sekcí.
- **EC-20-01-03 — DEK už byl shrednut před exportem (race erase vs export)** · Trigger: subjekt zažádal erase i export, erase doběhl první · Očekávané chování: šifrovaná pole se vrátí jako `[erased]` (ne crash, ne ciphertext) · Mechanismus: model-level konvertor vrací sentinel `[erased]` při chybějícím/shrednutém DEK (stejně jako `users.EmailHash`/audit) · Severity: P1 · Test: integrační — shred DEK, pak export → text chunků `[erased]`.
- **EC-20-01-04 — Soft-deleted dokument** · Trigger: `Document.IsDeleted == true` ale chunky stále existují · Očekávané chování: export MUSÍ zahrnout i soft-deleted dokumenty (uživatel má právo na všechna svá data, i „smazaná" ale fyzicky retenovaná) NEBO je explicitně označit `deleted_at` · Mechanismus: handler ignoruje `ISoftDeletable` query filtr přes `IgnoreQueryFilters()` jen pro tento export, jinak by data zmizela z exportu ale zůstala v DB (skrytá data = GDPR riziko) · Severity: P1 · Test: integrační — soft-delete doc → export stále obsahuje jeho chunky s flagem deleted.
- **EC-20-01-05 — Oversized export (subjekt má 500k chunků)** · Trigger: power-user s obrovským korpusem · Očekávané chování: export streamuje/pageuje, nenačítá vše do paměti najednou; nesmí OOM worker · Mechanismus: `AsAsyncEnumerable()` + dávkový zápis do exportního artefaktu (blob přes `IFileStorage`), ne jeden materializovaný list · Severity: P2 · Test: zátěžový — 100k chunků → konstantní paměť.
- **EC-20-01-06 — Embedding vektory v exportu** · Trigger: rozhodnutí zda 3072-dim vektory jsou osobní údaj · Očekávané chování: vektory jsou DERIVÁT obsahu; export zahrne zdrojový text (osobní údaj), vektory NE (nejsou srozumitelné subjektu a jsou rekonstruovatelné z textu) — zdokumentovat rozhodnutí · Mechanismus: DTO mapper vynechá `Embedding` sloupec; pozn. v doc · Severity: P3 · Test: assert export DTO nemá raw vektor.
- **EC-20-01-07 — Graph node sdílený mezi Tenant a User scope** · Trigger: entita grafu vznikla z user-private dokumentu i z tenant dokumentu (over-merge) · Očekávané chování: do osobního exportu jen ty `GraphNode` se `Scope=User && OwnerUserId=U`; tenant uzly se nevynáší · Mechanismus: striktní filtr na scope; pokud entita over-merged napříč scope → musí být dvě oddělené node (viz oblast graf), jinak leak · Severity: P1 · Test: integrační — tenant node se neobjeví v user exportu.
- **EC-20-01-08 — Cross-tenant izolace při exportu** · Trigger: uživatel je členem dvou tenantů (různé `OwnerUserId`? — ve skutečnosti `UserId` je per-identita) · Očekávané chování: export běží per-userId a NESMÍ obsáhnout data z jiného tenanta, kde tentýž člověk figuruje · Mechanismus: filtr `OwnerUserId == userId`; tenant data nikdy nejsou user-owned export · Severity: P0 · Test: integrační — dva tenanti, export jen jeden řez.

## UC-20-02 — Výmaz osobních dat uživatele (IErasePersonalData)
- **Actor / role:** system/worker (handler `UserErasureRequested`)
- **Precondition:** Subjekt `U` zažádal o výmaz; Gdpr modul publikoval `UserErasureRequested`; HybridRag má registrované `AddScoped<IErasePersonalData, HybridRagPersonalDataEraser>()`.
- **Trigger:** Worker handler `UserErasureRequested` fan-out volá `IErasePersonalData.EraseAsync(userId, ct)`.
- **Main flow:** handler dispatch `EraseHybridRagDataCommand(userId)` → otevře write kontext (`IDbContextOutbox`) → smaže/anonymizuje VŠECHNY `Chunk` (vč. `Embedding` + `SearchVector`), `Document` (anonymizuj metadata, smaž blob přes `IFileStorage.DeleteAsync`), `GraphNode`/`GraphEdge`/`EntityAlias` kde `OwnerUserId == U && Scope == User` → audit řádky se NEMAŽOU (append-only, retence pro AML/forenzní), místo toho je jejich PII chráněna krypto-shredem DEK (UC-20-03) → `SaveChangesAndFlushMessagesAsync` (commit) → idempotentní.
- **Postcondition / záruky:** Žádný user-owned chunk/vektor/graf uzel subjektu fyzicky nepřežije; blob bytes smazány; audit řádky retenovány ale PII v nich → `[erased]` (díky shredu DEK); operace idempotentní (druhé volání no-op). Tenant-scoped data subjektu se NEMAŽOU jen proto, že je vytvořil (firemní data; anonymizuje se autorství, ne obsah).
- **Tenancy / permissions:** Scope=User; SYSTEM kontext (worker vidí vše) → handler filtruje `OwnerUserId`. Žádná HTTP permission.
- **Reuse / canonical pattern:** Billing/Notifications `IErasePersonalData`; CLAUDE.md §4 „GDPR erasure" + „physically delete append-only ledger/audit rows … anonymize instead"; blob delete = `UploadFileHandler.cs:21` (compensating delete).
- **Data dotčená:** `chunks` (hard delete), `documents` (anonymizace/soft), `graph_nodes`/`graph_edges`/`entity_aliases`, blob storage · **Eventy:** žádný nový (reaguje na `UserErasureRequested`)
- **Priorita:** P0

- **EC-20-02-02 — Erase během běžícího ingest sagy** · Trigger: `IngestSaga` pro `U` je v poločase, erase přijde mezitím · Očekávané chování: saga buď doběhne a vytvoří chunky, které pak osiří (leak!), nebo erase musí saga abortnout · Mechanismus: erase handler MUSÍ označit subjekt jako „erasing" (tombstone/flag) a běžící saga při dalším kroku zkontroluje a self-abandon (`Abandoned`); po erase znovu-sweep, aby nezůstaly chunky vytvořené po prvním delete · Severity: P0 · Test: integrační — start ingest, erase mid-flight → 0 chunků nakonec, saga `Abandoned`.

- **EC-20-02-04 — Audit řádky se NESMÍ smazat** · Trigger: handler by se pokusil hard-delete `hybridrag_audit_entries` · Očekávané chování: audit zůstává, PII v něm je `[erased]` díky shredu · Mechanismus: erase NEMAŽE audit; spoléhá na krypto-shred DEK (UC-20-03); ArchUnitNET/review hlídá zákaz delete audit · Severity: P0 · Test: integrační — po erase audit řádky existují, PII pole `[erased]`.
- **EC-20-02-05 — Graph supernode sdílený více subjekty** · Trigger: `GraphNode` „Praha" referencovaný uzly více uživatelů · Očekávané chování: smaže se jen user-owned uzel/hrany subjektu; sdílené tenant uzly zůstávají; hrany vedoucí ze smazaného uzlu se smažou kaskádově (žádné dangling edges) · Mechanismus: delete `GraphEdge` kde `SourceNodeId ∈ smazané ∨ TargetNodeId ∈ smazané`; LINQ, ne raw SQL; tenant supernode nedotčen · Severity: P0 · Test: integrační — user node pryč, tenant supernode + jeho ostatní hrany intaktní, žádná hrana neukazuje na neexistující node.
- **EC-20-02-06 — Dangling edge po deletu (referenční integrita)** · Trigger: smazán node, hrana zůstane · Očekávané chování: žádná hrana nesmí ukazovat na neexistující node · Mechanismus: hrany se mažou ve stejné transakci před/spolu s nodes; ověření invariantu · Severity: P0 · Test: integrační — assert 0 hran s neexistujícím endpointem.
- **EC-20-02-07 — `ExecuteDelete` obchází audit/xmin** · Trigger: pokušení udělat hromadný `chunks.Where(...).ExecuteDeleteAsync()` · Očekávané chování: u erase je hromadný delete chunků OK (chunky nepotřebujeme auditovat při erase — mizí), ale audit entit (`Document` jako `IDataSubject`) NESMÍ jít přes `ExecuteDelete`/`ExecuteUpdate` pokud chceme změnu zaznamenat · Mechanismus: chunky/vektory → `ExecuteDelete` (perf, neauditované, záměrné jako ledger guard); `Document` anonymizace → tracked update aby vznikl audit záznam „erased" · Severity: P1 · Test: integrační — po erase je audit „document anonymized" přítomen.
- **EC-20-02-08 — Tenant-scoped data subjektu** · Trigger: subjekt nahrál firemní dokument (Scope=Tenant) · Očekávané chování: obsah se NEMAŽE (firemní vlastnictví), ale `OwnerUserId`/autorství se anonymizuje (nahradí systémovým/null) · Mechanismus: tracked update `OwnerUserId = null` na tenant dokumentech autora; obsah zůstává · Severity: P1 · Test: integrační — tenant doc přežije, autor odstraněn.

## UC-20-03 — Krypto-shred DEK subjektu (subject key)
- **Actor / role:** system/worker (Gdpr modul, navazuje na HybridRag erase)
- **Precondition:** Subjekt má DEK v `subject_keys` (Gdpr modul); HybridRag `Chunk.Content`/`GraphNode.PropsJson` jsou `[Encrypted]` pod tímto DEK.
- **Trigger:** `UserErasureRequested` → Gdpr `ShredSubjectKey(userId)` (existující mechanismus) po doběhnutí per-modul eraserů.
- **Main flow:** Gdpr eraser fan-out doběhne (vč. HybridRag UC-20-02) → Gdpr volá existující `ShredSubjectKey` → DEK se přepíše/odstraní, tombstone řádek v `subject_keys` zůstává PERMANENTNĚ (re-mint guard) → veškerý ciphertext šifrovaný tím DEK (audit PII, případné reziduální `[Encrypted]` hodnoty) je nenávratně nečitelný.
- **Postcondition / záruky:** Audit PII subjektu i jakýkoli neodstraněný `penc:v2` envelope → trvale `[erased]`; tombstone nesmí být nikdy smazán (jinak by post-erasure zápis re-mintnul čitelný klíč).
- **Tenancy / permissions:** SYSTEM; krytí Gdpr modulem; HybridRag jen poskytuje `IDataSubject` entity (`Document`).
- **Reuse / canonical pattern:** Gdpr `CryptoShredder` + `SubjectKey` + `ShredSubjectKey`; CLAUDE.md §4 „Audit-PII crypto-shred" + „Retention sweep" (tombstone retain permanently).
- **Data dotčená:** `subject_keys` (tombstone), nepřímo všechny `[Encrypted]` HybridRag hodnoty · **Eventy:** žádný nový
- **Priorita:** P0

## UC-20-04 — PII v klidu na Chunk.Content a GraphNode.PropsJson ([Encrypted][PersonalData])
- **Actor / role:** system/worker (ingest pipeline) zapisuje; user/tenant-admin čte přes retrieval
- **Precondition:** Ingestovaný dokument obsahuje PII (jména, e-maily, čísla); `Chunk.Content` má atributy `[Encrypted][PersonalData]`, `Document` implementuje `IDataSubject`; `GraphNode.PropsJson` má `[Encrypted]`.
- **Trigger:** ingest handler ukládá chunky → `PersonalDataEncryptionInterceptor` zašifruje při `SaveChanges`.
- **Main flow:** ingest handler vytvoří `Chunk` s plaintext `Content` → audit interceptor (pokud relevantní) → `PersonalDataEncryptionInterceptor` (běží PO auditu) zapečetí `Content` do `penc:v2` envelope (AES-GCM, AAD=subjectId) pod DEK subjektu (autora/ownera) → uloží ciphertext. Retrieval čte přes read factory → model-level konvertor dešifruje → reranker/LLM dostane plaintext jen za běhu.
- **Postcondition / záruky:** Žádný plaintext PII v `chunks.content`/`graph_nodes.props_json` sloupci na disku; AAD váže ciphertext na subjekt (zabrání záměně řádků); erasure shredem znečitelní.
- **Tenancy / permissions:** PII šifrována pod DEK ownera; tenant-scoped chunky pod tenant/ownerovým DEK (rozhodnout subjekt: u tenant scope je „subjekt" autor dokumentu).
- **Reuse / canonical pattern:** CLAUDE.md §4 „PII at rest (live columns)" `[Encrypted]` + `PersonalDataEncryptionInterceptor` + `users.EmailHash`; `User`/`Notification` jako `IDataSubject`.
- **Data dotčená:** `chunks.content`, `graph_nodes.props_json` · **Eventy:** žádný (interceptor)
- **Priorita:** P0

### Edge cases UC-20-04
