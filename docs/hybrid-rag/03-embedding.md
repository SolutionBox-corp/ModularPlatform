# Oblast 03 — Embedding generation
Pokrývá generování vektorových embeddingů pro chunky a dotazy přes `IEmbeddingGateway`/`IEmbeddingGenerator` (OpenAI `text-embedding-3-large`, 3072 dim), batchování, fake gateway pod `Rag:UseFakeGateways`, řešení provider chyb (429/down), dimenzní a model-drift kontrolu, MRL truncation, cost metriku a empty-text guard. Mapuje na build fázi **ingest pipeline** (chunk → embed → persist `Chunk.Embedding`) a **query** (dotaz → embed → vektorové retrieval) — embedding je sdílený stavební kámen obou. Embedding je deterministický side-effect externího providera, proto vše běží jako durable, idempotentní, perzistovaná jednotka práce; never fire-and-forget v HTTP requestu pro ingest.

## UC-03-01 — Vygenerování embeddingu jednoho chunku během ingestu (worker)
- **Actor / role:** system/worker
- **Precondition:** Existuje `Chunk` s neprázdným `Content` (+ volitelný `ContextualPrefix`), `Document.Status = Chunked`, kolekce existuje, embedding sloupec dosud `NULL`. `Rag:Embedding:Model = text-embedding-3-large`, `Rag:Embedding:Dimensions = 3072`.
- **Trigger:** Wolverine durable message `EmbedChunksCommand` (krok `IngestSaga`) doručená do Worker hostu.
- **Main flow:** `IngestSaga` publikuje `EmbedChunksCommand` → Worker handler (`EmbedChunksHandler`, public shell) → `dispatcher.Send(EmbedChunksCommand)` → handler načte `Chunk` přes `IDbContextOutbox<RagDbContext>.DbContext` → sestaví embed input `ContextualPrefix + "\n" + Content` → `IEmbeddingGateway.EmbedAsync(inputs, ct)` → vrácený `float[3072]` zapíše do `chunk.Embedding` (pgvector `Vector`) + nastaví `chunk.EmbeddingModel = "text-embedding-3-large"`, `chunk.EmbeddingDimensions = 3072`, `chunk.IsCurrent = true` → `SaveChangesAndFlushMessagesAsync` (commit) → saga posune stav na `Embedded`/`Indexed`.
- **Postcondition / zaruky:** `chunk.Embedding` perzistován, xmin token aktualizován, audit zapsán (změna `Embedding`/`EmbeddingModel` přes `AuditInterceptor`). Idempotence: opakované doručení téhož `EmbedChunksCommand` přepíše stejnou hodnotou (deterministicky stejný model+text → stejný vektor u fake; u OpenAI se přijme nová hodnota, ale stav `IsCurrent` zůstává konzistentní). Žádný event navenek — interní krok pipeline.
- **Tenancy / permissions:** Scope dědí z `Chunk` (`ITenantScoped` + `IUserOwned` přes `OwnerUserId`), worker běží jako SYSTEM (`SystemTenantContext` nebo `HttpTenantContext` bez HttpContextu), ale zápis respektuje RLS GUC `app.principal_id` nastavený dle `OwnerUserId` chunku. Žádná uživatelská permission (interní).
- **Reuse / canonical pattern:** Worker handler shell `ProvisionCreditAccountHandler.cs:13`; outbox commit `RegisterUserHandler.cs:22`; saga krok `CreditPurchaseSaga.cs:30`.
- **Data dotcena:** `chunks` (Embedding, EmbeddingModel, EmbeddingDimensions, IsCurrent) · **Eventy:** žádný (interní saga krok)
- **Priorita:** P0

### Edge cases UC-03-01
- **EC-03-01-01 — Prázdný/whitespace `Content`** · Trigger: chunk s `Content = ""`/jen mezery (degenerovaný chunking) · Ocekavane chovani: NEVOLAT providera (OpenAI vrací 400 na empty input + plýtvá cost); chunk se buď přeskočí a označí `EmbeddingSkipped = true`, nebo saga selže s `BusinessRuleException("rag.embed.empty_input")`. Preferováno: tiše přeskočit + WARN log + metrika `platform.rag.embed_skipped_empty`, NIKDY uložit nulový vektor jako validní. · Mechanismus: guard v handleru před `EmbedAsync`, validace; Zákon „graceful degradation = explicit flag, ne tichá půlka" · Severity: P1 · Test: integ — chunk s prázdným contentem → fake gateway nevolán (spy), chunk má `EmbeddingSkipped`, saga pokračuje.
- **EC-03-01-02 — Token-window overflow (chunk > 8191 tokenů modelu)** · Trigger: `ContextualPrefix + Content` přesáhne max input tokens `text-embedding-3-large` (8191) · Ocekavane chovani: detekovat délku PŘED voláním (tokenizer/heuristika znaky→tokeny) a buď bezpečně zkrátit na limit, nebo selhat s `BusinessRuleException("rag.embed.token_overflow")` — preferováno tvrdé selhání, protože chunking měl garantovat velikost (overflow = bug výše). NIKDY nechat OpenAI vrátit 400 a zabít sagu bez kontextu. · Mechanismus: pre-flight token count guard ve handleru; chunking invariant je vlastník velikosti · Severity: P1 · Test: integ — uměle nafouknutý chunk → handler vyhodí typed exception s errorCode, saga jde do retry/abandon dle politiky.
- **EC-03-01-03 — OpenAI 429 (rate limit) s `Retry-After`** · Trigger: gateway dostane HTTP 429 + header `Retry-After: N` · Ocekavane chovani: respektovat `Retry-After`, nepokoušet se okamžitě znovu; vyhodit retryable exception → Wolverine retry-with-cooldown ji znovu doručí po cooldownu. NIKDY busy-loop. · Mechanismus: `IEmbeddingGateway` mapuje 429 na `EmbeddingRateLimitedException` (retryable) s `RetryAfter`; `PlatformMessaging` retry-with-cooldown → durable dead-letter po vyčerpání; Polly/`p-retry` ekvivalent uvnitř gateway pro krátké backoffy · Severity: P0 · Test: integ — fake gateway nastaven vyhodit 429 jednou → po retry uspěje, chunk embeddován; assert počet pokusů ≥ 2.
- **EC-03-01-04 — OpenAI provider down (5xx / timeout / DNS)** · Trigger: 500/502/503/504 nebo `TaskCanceledException` z HTTP klienta · Ocekavane chovani: retryable, exponenciální backoff přes Wolverine; po vyčerpání retry → durable dead-letter (NE ztráta chunku), saga zůstane v `Embedding`, `MessagingHealthJob` to zachytí jako stuck. · Mechanismus: gateway → `EmbeddingProviderUnavailableException` (retryable); `PlatformMessaging` retry/DLQ; alert metrika `platform.messaging.dead_letters` · Severity: P0 · Test: integ — fake vyhodí 503 trvale → po N retry chunk v DLQ, `Document.Status` zůstane `Chunked`/`Embedding`, žádný falešný `Indexed`.
- **EC-03-01-05 — HTTP timeout uprostřed (vektor možná spočítán na straně OpenAI, ale odpověď ztracena)** · Trigger: klient timeout po odeslání, OpenAI to zaúčtoval · Ocekavane chovani: idempotentní retry — opětovné volání je bezpečné (embedding je čistá funkce textu), cost se zdvojí (přijatelné), ale stav konzistentní. NIKDY částečný zápis. · Mechanismus: `SaveChangesAndFlushMessagesAsync` je atomický commit; retry přepíše stejný chunk · Severity: P2 · Test: integ — timeout na první pokus, druhý OK → přesně jeden zápis Embeddingu, IsCurrent=true.
- **EC-03-01-06 — Concurrent embed zápis na stejný chunk (duplicitní doručení / paralelní worker)** · Trigger: dvě instance Workeru zpracují tutéž zprávu (Balanced mód, competing consumers) · Ocekavane chovani: inbox dedup (UNIQUE MessageId) zahodí duplikát; pokud projdou obě, xmin + `ConcurrencyRetryBehavior` serializuje, jeden uspěje, druhý retry → no-op (stejná hodnota). NIKDY torn write vektoru. · Mechanismus: Wolverine inbox dedup; xmin na `Chunk` (pozn.: `Chunk : Entity` má xmin) + `ConcurrencyRetryBehavior` · Severity: P1 · Test: integ 2× paralelní `EmbedChunksCommand` se stejným MessageId → jeden zápis, žádná výjimka nahoru.
- **EC-03-01-07 — Dimenze z providera ≠ 3072** · Trigger: OpenAI vrátí vektor jiné délky (změna API, špatná konfigurace `dimensions` param) · Ocekavane chovani: HARD FAIL před zápisem — `BusinessRuleException("rag.embed.dimension_mismatch")`; NIKDY uložit vektor jiné dimenze do `vector(3072)` sloupce (pgvector by stejně odmítl, ale chceme čistou typed chybu, ne DbUpdateException). · Mechanismus: guard `if (vec.Length != ExpectedDimensions) throw` ve handleru; sloupcová dimenze pgvector jako poslední pojistka · Severity: P0 · Test: unit — fake vrátí délku 1536 → typed exception, žádný DB zápis.
- **EC-03-01-08 — Soft-deleted dokument mezitím** · Trigger: `Document.IsDeleted = true` (smazán uživatelem) než doběhne embed · Ocekavane chovani: přeskočit embed (zbytečná práce + stale index), saga ukončit jako `Canceled`/`Skipped`; chunk se neembedduje. · Mechanismus: re-fetch `Document` v handleru, kontrola `ISoftDeletable`/global query filter; idempotentní krok · Severity: P2 · Test: integ — označit doc deleted, doručit embed → fake nevolán, chunk bez Embeddingu, saga terminálně Skipped.
- **EC-03-01-09 — Saga crash mezi `EmbedAsync` a commitem** · Trigger: worker spadne po obdržení vektoru, před `SaveChangesAndFlushMessagesAsync` · Ocekavane chovani: žádný commit → zpráva se po restartu znovu doručí (durable inbox), embed proběhne znovu, idempotentně. · Mechanismus: transakční outbox — buď commit + flush, nebo nic; Wolverine resume · Severity: P1 · Test: integ — simulovaný crash (throw po EmbedAsync) → po re-deliveru chunk korektně embeddován jednou.
- **EC-03-01-10 — Non-UTF8 / control znaky v contentu** · Trigger: chunk obsahuje neplatné/binární sekvence (špatná extrakce dokumentu) · Ocekavane chovani: sanitizace (strip control chars kromě \n\t) před voláním; pokud po sanitizaci prázdné → EC-03-01-01. NIKDY nepropustit raw bytes do JSON requestu (rozbije serializaci). · Mechanismus: gateway/handler normalizace inputu; battle-tested encoding handling (nepsat vlastní) · Severity: P2 · Test: unit — vstup s \u0000 → sanitizován, volání proběhne s čistým textem.

## UC-03-02 — Batch embedding mnoha chunků jedním voláním providera
- **Actor / role:** system/worker
- **Precondition:** Dokument rozdělen do N chunků (N může být stovky), všechny čekají na embed. `Rag:Embedding:BatchSize` (default např. 96) konfiguruje velikost dávky.
- **Trigger:** `EmbedChunksCommand` nesoucí seznam `ChunkId[]` (nebo saga iteruje po dávkách).
- **Main flow:** Handler načte dávku chunků → sestaví pole inputů (zachová pořadí) → `IEmbeddingGateway.EmbedBatchAsync(inputs[], ct)` (OpenAI Embeddings API přijímá pole) → vrácené vektory namapuje 1:1 podle indexu zpět na `ChunkId` → bulk update tracked entit → `SaveChangesAndFlushMessagesAsync`. Velké N rozděleno do více volání po `BatchSize`.
- **Postcondition / zaruky:** Všechny chunky dávky mají Embedding nebo dávka celá selže (atomický commit dávky); cost metrika `platform.rag.embed_tokens` inkrementována o sumu tokenů. Idempotence per chunk přes opětovné doručení.
- **Tenancy / permissions:** Všechny chunky v dávce MUSÍ patřit témuž `OwnerUserId`/Scope (RLS GUC je per-connection jediná hodnota) — dávkování je per-owner. SYSTEM kontext.
- **Reuse / canonical pattern:** `ProvisionCreditAccountHandler.cs:13` (handler shell); batch pattern vlastní, ale commit `RegisterUserHandler.cs:22`.
- **Data dotcena:** `chunks` (bulk Embedding) · **Eventy:** žádný
- **Priorita:** P0

### Edge cases UC-03-02
- **EC-03-02-01 — Mismatch počtu vrácených vektorů vs. inputů** · Trigger: OpenAI vrátí méně/více embeddingů než vstupů (anomálie) · Ocekavane chovani: HARD FAIL celé dávky `BusinessRuleException("rag.embed.batch_count_mismatch")`, žádný částečný zápis (jinak by se vektory namapovaly na špatné chunky = katastrofa retrievalu). · Mechanismus: guard `outputs.Length == inputs.Length`; mapování striktně podle `index` z odpovědi, ne podle pořadí pole · Severity: P0 · Test: unit — fake vrátí N-1 vektorů → exception, 0 zápisů.
- **EC-03-02-02 — Špatné pořadí ve výsledku (OpenAI `index` pole)** · Trigger: odpověď přijde v jiném pořadí než input · Ocekavane chovani: mapovat podle pole `index` v response, NIKDY spoléhat na pořadí pole (out-of-order safe princip). · Mechanismus: handler řadí výsledky podle `data[].index` před mapováním na ChunkId · Severity: P0 · Test: unit — fake vrátí přeházené pořadí → každý chunk dostane SVŮJ vektor (assert deterministicky podle obsahu).
- **EC-03-02-03 — Batch token limit překročen (suma tokenů dávky > limit modelu)** · Trigger: dávka 96 dlouhých chunků překročí per-request token cap OpenAI · Ocekavane chovani: dynamické sub-dávkování podle odhadu tokenů (ne jen podle počtu), nebo retry s menší dávkou po 400. NIKDY zafixovat jen na počtu položek. · Mechanismus: token-aware batching ve handleru/gateway; fallback split při 400 `max_tokens_per_request` · Severity: P1 · Test: integ — dávka přes token limit → automaticky rozdělena na 2 volání, všechny chunky embeddovány.
- **EC-03-02-04 — Částečné 429 uprostřed multi-batch sekvence** · Trigger: dokument má 500 chunků = 6 volání, 4. vrátí 429 · Ocekavane chovani: dávky 1-3 commitnuté (per-batch commit), 4. retry s `Retry-After`, pak pokračovat 5-6. Idempotence: re-embed už hotových chunků je no-op/přepis. NIKDY ztratit progres. · Mechanismus: per-batch commit + Wolverine retry; saga drží cursor odembeddovaných · Severity: P1 · Test: integ — 429 na 4. dávku → po retry kompletní embedding všech 500.
- **EC-03-02-05 — Jeden chunk v dávce prázdný** · Trigger: 1 z 96 inputů je prázdný · Ocekavane chovani: vyfiltrovat prázdné PŘED odesláním (jinak OpenAI 400 zabije celou dávku), prázdné označit `EmbeddingSkipped`, zbytek embeddovat. · Mechanismus: pre-filter inputů + zachování indexové mapy na původní ChunkId · Severity: P1 · Test: integ — dávka s 1 prázdným → 95 embeddováno, 1 skip, žádné 400.
- **EC-03-02-06 — Supernode / extrémně velká dávka v paměti** · Trigger: 10k chunků najednou → 10k × 3072 × 4B ≈ 120 MB float v RAM · Ocekavane chovani: streamovat/dávkovat, nikdy nedržet všechny vektory v jednom poli; commit po dávkách uvolní paměť. · Mechanismus: bounded batch size + per-batch commit · Severity: P2 · Test: load — dokument 10k chunků projde bez OOM (sledovat peak RAM).
- **EC-03-02-07 — Cost metrika přesnost při retry** · Trigger: dávka retryována 3× kvůli 429 · Ocekavane chovani: cost/token metrika počítá KAŽDÉ skutečné volání providera (i neúspěšná, pokud byla zaúčtována), aby observabilita nelhala o nákladech. · Mechanismus: inkrement `platform.rag.embed_tokens`/`platform.rag.embed_requests` v gateway na úrovni HTTP volání, ne handleru · Severity: P2 · Test: integ — 3 pokusy → metrika requests = 3, ne 1.

## UC-03-03 — Embedding dotazu (query-time) pro retrieval
- **Actor / role:** user | tenant-admin (přes search endpoint)
- **Precondition:** Uživatel autentizován, zadá `query` string; retrieval slice potřebuje query vektor pro pgvector `CosineDistance`.
- **Trigger:** Synchronní volání uvnitř search query handleru (`HybridSearchQuery`) — NE durable zpráva (query je read-only, low-latency).
- **Main flow:** Search handler → `IEmbeddingGateway.EmbedAsync([query], ct)` (single input) → `float[3072]` → použit v LINQ `chunks.OrderBy(c => c.Embedding.CosineDistance(queryVec)).Take(k)`. Query embedding se NEPERZISTUJE (cache-able krátkodobě, ale není entita).
- **Postcondition / zaruky:** Query vektor použit, žádný DB zápis, žádný event (query = jen čtení, NIKDY transakce). Latence kritická — embed je v hot path.
- **Tenancy / permissions:** Embed sám je tenant-agnostický (jen text→vektor), ale následný retrieval je RLS/Scope filtrovaný. Žádná zvláštní permission nad search permission.
- **Reuse / canonical pattern:** Read query `GetProfileHandler.cs:12` (`IReadDbContextFactory`); query NIKDY neotevírá transakci (Zákon).
- **Data dotcena:** žádná zapisovaná (jen čte `chunks` v retrievalu) · **Eventy:** žádný
- **Priorita:** P0

### Edge cases UC-03-03
- **EC-03-03-01 — Prázdný dotaz** · Trigger: `query = ""`/whitespace · Ocekavane chovani: validace `HybridSearchValidator` odmítne PŘED embedem → `ValidationException("rag.search.query_empty")`, žádné volání providera. · Mechanismus: FluentValidation `.NotEmpty().WithErrorCode("rag.search.query_empty")` · Severity: P1 · Test: integ — prázdný query → 400 RFC9457, gateway nevolán.
- **EC-03-03-02 — Provider down při query** · Trigger: OpenAI 5xx/timeout během query embedu · Ocekavane chovani: GRACEFUL DEGRADATION — vrátit výsledek označený `Degraded = true` s fallbackem na čistě lexikální BM25 (`pg_search`; `ts_rank_cd` fallback) bez vektorové části, NEBO 503 `BusinessRuleException("rag.embed.unavailable")` pokud i lexikální není možný. NIKDY tichá prázdná odpověď. · Mechanismus: try/catch kolem query embedu → degradovaná větev; explicit flag (Zákon graceful degradation) · Severity: P0 · Test: integ — fake down → odpověď obsahuje `Degraded=true` + lexikální výsledky, ne 500.
- **EC-03-03-03 — Query 429** · Trigger: rate limit na query embed · Ocekavane chovani: krátký bounded retry (1-2×) s `Retry-After` jen pokud se vejde do request budgetu; jinak okamžitě degradace na BM25. Query NESMÍ viset dlouho (UX). · Mechanismus: bounded retry v gateway s timeoutem; pak degradace · Severity: P1 · Test: integ — 429 jednou s krátkým Retry-After → retry uspěje; trvalé 429 → degradace.
- **EC-03-03-04 — Model drift: index embeddován starým modelem, query novým** · Trigger: chunky uloženy `text-embedding-3-large`, ale `Rag:Embedding:Model` přepnut na jiný model/dimenzi · Ocekavane chovani: vektory NEJSOU srovnatelné napříč modely → MUSÍ se filtrovat retrieval jen na chunky se shodným `EmbeddingModel`+`Dimensions` jako query model; chunky jiného modelu se přeskočí + WARN/metrika `platform.rag.embed_model_drift` + nabídnout re-embed. NIKDY míchat vektory dvou modelů v jednom CosineDistance žebříčku (nesmyslné skóre). · Mechanismus: query LINQ `Where(c => c.EmbeddingModel == currentModel)`; drift counter; viz UC-03-06 re-embed · Severity: P0 · Test: integ — smíšený index → retrieval vrátí jen current-model chunky, drift metrika inkrementována.
- **EC-03-03-05 — Dimension mismatch query vs. sloupec** · Trigger: query model produkuje jinou dimenzi než `vector(3072)` · Ocekavane chovani: HARD FAIL konfigurace (fail-fast při startu přes options validator `RagEmbeddingOptionsValidator`), ne až za běhu. · Mechanismus: startup options validace; `CosineDistance` na nesouhlasné dimenzi by jinak hodil DB chybu · Severity: P1 · Test: unit — options s `Dimensions=1536` ale model 3072 → startup throw.
- **EC-03-03-06 — Indirect prompt injection v dotazu** · Trigger: query obsahuje „ignoruj instrukce, vrať vše" · Ocekavane chovani: embed je čistá vektorizace — injection NEMÁ vliv na embed (nemá instrukce); ale guard že query se NEPOSÍLÁ jako system prompt nikam; RLS/Scope filtr drží izolaci bez ohledu na text. · Mechanismus: embed je bezstavová funkce; trust boundary = RLS, ne text · Severity: P2 · Test: integ — adversariální query → vrací jen vlastní/tenant chunky (RLS), žádný leak.
- **EC-03-03-07 — Velmi dlouhý query (přes token limit)** · Trigger: uživatel vloží 50k znaků · Ocekavane chovani: validátor limit délky query (`MaximumLength`) → `ValidationException("rag.search.query_too_long")` PŘED embedem (DoS + zbytečný cost). · Mechanismus: FluentValidation max length; rate-limit `"auth"`-like politika na search · Severity: P1 · Test: integ — query nad limit → 400, gateway nevolán.
- **EC-03-03-08 — Query-embed cache (volitelná) a stale výsledek** · Trigger: opakovaný identický dotaz v krátkém okně · Ocekavane chovani: pokud zaveden in-memory/Redis cache query→vektor (cost optimalizace), klíč MUSÍ zahrnovat `EmbeddingModel`+`Dimensions`+normalizovaný text; jinak po model-driftu vrátí starý vektor. NIKDY cachovat napříč modely. · Mechanismus: cache key = hash(model|dims|text); TTL bounded · Severity: P2 · Test: unit — změna modelu invaliduje cache (jiný klíč).

## UC-03-04 — Fake embedding gateway pod `Rag:UseFakeGateways` (testy)

- **Main flow:** `FakeEmbeddingGateway.EmbedAsync(text)` → deterministicky: hash textu (SHA-256) → seed → naplní `float[3072]` pseudonáhodně + L2-normalizuje. Stejný text → stejný vektor (deterministická testovatelnost retrievalu). Žádné HTTP, žádný cost.
- **Postcondition / zaruky:** Deterministické vektory umožní stabilní asserty retrievalu (podobné texty → blízké vektory pokud fake zachovává nějakou strukturu, jinak alespoň identita). Žádné externí volání.

- **Reuse / canonical pattern:** Fake-under-flag `MarketingModule.cs:51` (`Marketing:UseFakeGateways`); `FakeStripeGateway` pattern (Billing).
- **Data dotcena:** žádná · **Eventy:** žádný
- **Priorita:** P0
